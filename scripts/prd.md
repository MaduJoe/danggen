# PRD: '당근' E2E 테스트 자동화 프레임워크 구축

## 1.0 서론 및 전략적 배경

### 1.1. 프로젝트 비전
본 프로젝트의 비전은 하이퍼로컬 플랫폼 '당근' 애플리케이션을 위한 견고하고 확장 가능하며 다층적인 E2E(End-to-End) 테스트 자동화 프레임워크를 설계하고 구현하는 것입니다. 이 프레임워크는 단순한 테스트 스크립트의 집합이 아니라, 실제 운영 환경 수준의 품질을 갖춘 개념 증명(Proof-of-Concept)으로서 기능합니다. 모바일, API, 웹 인터페이스 전반에 걸친 핵심 사용자 여정을 검증하여 기능적 무결성, 성능, 그리고 끊김 없는 사용자 경험을 보장하는 것을 목표로 합니다. 최종적으로, 이 프레임워크는 현대적인 CI/CD 기반 개발 수명주기에서 핵심적인 역할을 수행하는 전략적 자산으로 설계됩니다.

### 1.2. 문제 정의 및 비즈니스 타당성

**도전 과제:**
'당근'은 대한민국을 대표하는 하이퍼로컬 커뮤니티 플랫폼으로서, 단순한 테스트 접근법으로는 검증하기 어려운 독특하고 복잡한 품질 보증(QA) 과제를 안고 있습니다. 주요 도전 과제는 다음과 같습니다.

- **하이퍼로컬 의존성:** 핵심 기능 대부분이 GPS 기반의 '동네인증'과 밀접하게 연관되어 있습니다. 이는 다양한 테스트 환경에서 각기 다른 위치 기반 시나리오를 일관성 있게 실행하고 재현하는 것을 기술적으로 어렵게 만듭니다.
- **실시간 비동기 상호작용:** '당근채팅'과 같은 기능은 다수의 사용자 간에 실시간으로 발생하는 비동기적 통신을 기반으로 합니다. 이러한 시나리오는 자동화하고 안정적으로 검증하기가 매우 까다로운 대표적인 영역입니다.
- **복잡하게 연결된 서비스 구조:** 플랫폼은 중고거래, 동네생활, 당근페이, 비즈프로필 등 수많은 서비스가 유기적으로 통합되어 운영됩니다. 한 서비스의 장애가 다른 서비스에 연쇄적인 영향을 미칠 수 있으므로, 전체적인 관점의 E2E 테스트 전략이 필수적입니다.
- **AI/ML 기반 기능:** '당근'은 TensorFlow와 같은 머신러닝 기술을 활용하여 사용자 맞춤형 콘텐츠 추천 및 유해 게시물 필터링 기능을 제공합니다. 이러한 비결정론적 시스템을 테스트하기 위해서는 단순한 입력-출력 검증을 넘어서는 고도화된 전략이 요구됩니다.

**해결 방안:**
이러한 리스크를 효과적으로 관리하고 완화할 수 있는 유일하고 현실적인 해결책은 전용 자동화 프레임워크를 구축하는 것입니다. 이 프레임워크는 시간이 많이 소요되고 사람의 실수에 취약한 수동 회귀 테스트를 대체하여 개발 과정에서 신속한 피드백 루프를 가능하게 합니다. 이를 통해 테스트 커버리지를 획기적으로 늘리고, QA 엔지니어는 보다 가치가 높은 탐색적 테스팅 및 품질 전략 수립에 집중할 수 있게 됩니다. 이는 과거 프로젝트에서 수동 리그레션 테스트 시간을 168시간에서 2시간으로 단축하며 98.8%의 효율성 향상을 달성한 경험과 직접적으로 맞닿아 있습니다.

### 1.3. 목표 직무와의 전략적 연계성
본 프로젝트는 목표로 하는 테스트 자동화 엔지니어 직무의 자격요건과 요구 역량을 구체적이고 실증적인 결과물로 증명하기 위해 명시적으로 설계되었습니다. 특히, 목표 직무에서 요구하는 "자기 주도적으로 일하는 것을 즐기고, 동료들과 커뮤니케이션이 원활하신 분"이라는 역량은 기술적 능력만큼이나 중요합니다. 잘 구조화되고, 명확하며, 포괄적인 내용을 담은 본 PRD(Product Requirements Document) 문서는 단순한 계획서를 넘어섭니다. 이 문서 자체가 복잡한 정보를 체계적으로 정리하고, 기술적 아이디어를 다양한 이해관계자에게 효과적으로 전달하는 커뮤니케이션 능력을 직접적으로 증명하는 산출물입니다.

시니어 직무에서는 주어진 업무를 수행하는 것을 넘어, 업무를 스스로 정의하고, 잠재적 문제를 예측하며, 기술적 해결책을 비즈니스 목표에 맞게 정렬하는 능력이 요구됩니다. 이 PRD는 이러한 자기 주도성과 전략적 사고 능력을 입증하는 핵심적인 증거 자료로 기능합니다.

## 2.0 대상 페르소나 및 핵심 사용자 여정
이 섹션에서는 프레임워크의 사용자를 정의하고, 자동화 테스트가 검증할 애플리케이션의 핵심 워크플로우를 구체적으로 명시합니다.

### 2.1. 프레임워크 사용자 페르소나: QA 자동화 엔지니어
이 프레임워크의 주 사용자는 '당근'의 중급-시니어 수준 QA 자동화 엔지니어입니다. 이들의 목표는 신뢰성 있는 자동화 테스트를 신속하게 작성, 실행, 그리고 디버깅하는 것이며, 유지보수가 용이하고 확장 가능한 프레임워크를 필요로 합니다.

### 2.2. 자동화를 위한 '당근' 핵심 사용자 여정
아래 사용자 여정들은 '당근' 플랫폼의 가장 중요하고, 트래픽이 많으며, 기술적으로 검증이 까다로운 기능들을 종합적으로 커버하도록 선정되었습니다.

#### 2.2.1. 여정 1: 핵심 중고거래 흐름 (정상 시나리오)
- **사전 준비 (Setup):** API를 통해 테스트에 사용할 두 명의 사용자(판매자, 구매자)를 프로그래밍 방식으로 생성합니다.
- **판매자 활동 (API):** 판매자 계정으로 로그인하여 '중고거래' 물품을 신규 등록합니다.
- **구매자 활동 (Mobile UI):** 구매자 계정으로 모바일 앱에 로그인하고, '동네인증'을 통해 위치를 확인한 후, 등록된 물품을 검색하여 피드에 정상적으로 노출되는지 확인합니다.
- **상호작용 (Mobile UI):** 구매자는 물품 상세 페이지로 진입하여 판매자에게 '당근채팅'을 시작합니다.
- **협상 (API/UI):** 간단한 채팅 교환을 자동화합니다 (예: 구매자 "구매 가능한가요?", 판매자 "네, 가능합니다.").
- **결제 (UI - 모의 처리):** 구매자가 '당근페이'를 통해 거래를 시작하며, 최종 결제 단계는 모의(Mock) 처리하거나 결제 확인 화면에서 테스트를 종료합니다.
- **사후 처리 (Teardown):** API를 통해 생성했던 사용자 및 게시물을 삭제합니다.

#### 2.2.2. 여정 2: 커뮤니티 참여 흐름
- **사전 준비 (Setup):** 특정 '동네'에 위치한 사용자를 생성합니다.
- **활동 (Mobile UI):** 사용자는 '동네생활' 탭에서 새로운 게시글을 작성합니다.
- **검증 (API):** API를 사용하여 게시글 목록에서 새로 작성된 게시글을 확인합니다.
- **상호작용 (UI):** 두 번째 사용자가 해당 게시글에 댓글을 작성합니다.
- **검증 (UI/API):** 원래 사용자가 게시글을 새로고침하여 댓글이 정상적으로 표시되는지 확인합니다.

#### 2.2.3. 여정 3: 지역 비즈니스 발견 흐름
- **사전 준비 (Setup):** 테스트 데이터베이스에 특정 지역 비즈니스의 '비즈프로필'이 존재하는지 확인합니다.
- **활동 (Mobile UI):** 사용자가 '동네' 내에서 키워드로 검색을 수행합니다.
- **검증 (UI):** 해당 비즈프로필이 검색 결과와 '동네지도'에 올바르게 나타나는지 검증합니다.
- **상호작용 (UI):** 사용자가 프로필을 클릭하여 상세 정보가 정확하게 표시되는지 확인합니다.

#### 2.2.4. 여정 4: 예외 경로 및 엣지 케이스 시나리오
- **금지 물품 등록 (ML 검증):** 정책을 위반하는 물품 등록을 시도하고, API를 통해 해당 게시물이 검토 상태로 변경되거나 등록 거부되는지를 검증합니다.
- **GPS 위치 조작:** 인증되지 않은 동네에서 활동을 시도하고, 애플리케이션이 예상된 오류를 표시하며 활동을 차단하는지 검증합니다.
- **결제 실패:** '당근페이' 흐름 중 오류를 시뮬레이션하여, UI에 사용자 친화적인 오류 메시지가 표시되고 거래 상태가 롤백되는지 검증합니다.

'당근'의 성공은 '신뢰'를 기반으로 하며, '매너온도'나 AI 필터링 시스템 같은 기능은 신뢰 유지를 위한 핵심 인프라입니다. 따라서 QA 전략은 이러한 신뢰 및 안전 메커니즘을 테스트하는 것을 최우선 과제로 삼아야 합니다.

## 3.0 프레임워크 기능 및 아키텍처
이 섹션은 프로젝트의 기술적인 방법론을 상세히 설명합니다.

### 3.1. 기본 아키텍처: 다층 모델 (Multi-Layered Model)
프레임워크는 관심사의 분리, 재사용성, 유지보수성을 극대화하기 위해 모듈화된 다층 아키텍처를 기반으로 구축됩니다.

- **1계층: 테스트 오케스트레이션 (Pytest):** Pytest를 핵심 테스트 실행기로 사용합니다.
- **2계층: 애플리케이션 드라이버 (Appium & Playwright):** Appium은 모바일 UI 자동화를, Playwright는 웹 기반 컴포넌트 자동화를 담당합니다.
- **3계층: 페이지 객체 모델 (Page Object Models, POM):** 모든 UI 상호작용은 페이지 객체 내에 캡슐화하여 테스트 로직과 UI 구현을 분리합니다.
- **4계층: API 클라이언트:** 백엔드 상호작용을 위한 전용 Python 클라이언트를 구현하여 테스트 데이터 생성/삭제 및 빠른 검증에 사용합니다.
- **5계층: 테스트 데이터 관리:** 테스트 실행에 필요한 데이터를 동적으로 생성, 관리 및 정리하는 모듈입니다.

### 3.2. 고급 자동화 기능

#### 3.2.1. 위치정보 모의(Mocking)
- **문제:** 다양한 사용자 위치 시뮬레이션 필요.
- **해결책:** Appium의 `driver.set_location` 기능을 통합하여 테스트 중 디바이스의 GPS 좌표를 동적으로 설정합니다. Pytest 픽스처를 통해 관리되며, `@pytest.mark.location('Yongsan')`과 같이 쉽게 파라미터화할 수 있습니다.

#### 3.2.2. 실시간 채팅 검증
- **문제:** 두 클라이언트의 동시 상호작용 검증 필요.
- **해결책:** 두 개의 Appium 드라이버 세션(발신자, 수신자)을 동시에 오케스트레이션하여 한 클라이언트가 보낸 메시지가 다른 클라이언트에게 정상적으로 수신되는지 검증합니다.

#### 3.2.3. 시각적 회귀 테스트
- **문제:** 기능은 정상이나 시각적 버그(예: 레이아웃 깨짐) 발생 가능.
- **해결책:** 스냅샷 테스팅 전략을 도입합니다. Playwright의 내장 기능이나 Percy 같은 라이브러리를 활용하여 기준 스크린샷과 테스트 중 촬영된 스크린샷을 비교해 의도치 않은 시각적 변화를 자동으로 감지합니다.

### 3.3. CI/CD 통합 및 리포팅

#### 3.3.1. 파이프라인 오케스트레이션 (GitHub Actions)
- `workflow.yml` 파일을 작성하여 CI 파이프라인을 정의합니다.
- **트리거:** Pull Request의 push 이벤트 및 main 브랜치 대상 야간 정기 실행으로 구성합니다.
- **매트릭스 빌드:** 빌드 매트릭스를 사용하여 다양한 구성에서 병렬로 테스트를 실행합니다.

#### 3.3.2. 종합 리포팅 (Allure)
- **문제:** 원시 테스트 로그는 분석이 어려워 이해관계자를 위한 고수준의 시각 자료가 필요합니다.
- **해결책:** Allure Framework와 통합하여 각 CI 실행 후 상호작용이 가능한 리포트를 생성합니다. 이 리포트는 대시보드, 상세 테스트 단계, 실패 시 스크린샷 및 비디오, 실행 환경 정보 등을 포함합니다.
